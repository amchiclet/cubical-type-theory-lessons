Why This Matters: The Payoff

One worked example, in two worlds. What univalence buys you, and what it costs to live without it.

The win for this lesson You can explain, on a concrete example, the single sentence at the centre of this whole subject: prove a property once, then move it across an equivalence for free. And you can say precisely what the world without univalence has to do instead.

The one-sentence thesis

Three parts, and it helps to keep them separate:

PartWhat it isNew?
Enginetransport: given a path A = B and a proof P(A), get P(B). Uniform over every predicate, via path induction.Old. Plain type theory already has this.
FuelUnivalence: turn an equivalence A ≃ B into a path A = B. Ordinary type theory has almost no way to prove two types equal.New. This is the HoTT move.
CatchIn book-HoTT univalence is an axiom with no computation rule, so the transported result gets stuck. Cubical makes it compute.The reason cubical exists.

The example: Bool and Binary

Two types that are obviously the same shape, just relabelled.

type Bool   = true | false
type Binary = zero | one

to   : Bool -> Binary          from : Binary -> Bool
to true  = one                 from one  = true
to false = zero                from zero = false
-- to and from are inverse, so we have an equivalence  e : Bool ≃ Binary

We will move a whole structure with its proofs, not just a value. Package "a type equipped with an idempotent operation" as a record:

Meet A = record {
  op   : A -> A -> A ,
  idem : (x : A) -> op x x = x
}

Now do the real work, once, on Bool:

and : Bool -> Bool -> Bool
and true  y = y
and false _ = false

andIdem : (x : Bool) -> and x x = x     -- proved by checking both cases
andIdem true  = refl
andIdem false = refl

boolMeet : Meet Bool
boolMeet = record { op = and , idem = andIdem }

World A — with univalence

Univalence turns the equivalence into a path p : Bool = Binary. Then one line hands you the whole structure on Binary, proof included:

p : Bool = Binary
p = ua e                                -- univalence: equivalence ⟶ path

binaryMeet : Meet Binary
binaryMeet = transport Meet p boolMeet  -- one line. no new proofs.

What you get, unfolded:

binaryMeet.op a b = to (and (from a) (from b))
   -- e.g.  binaryMeet.op one one = to (and true true) = to true = one

binaryMeet.idem : (a : Binary) -> binaryMeet.op a a = a
   -- ^ this PROOF was handed to you, for free. You never touched Binary's cases.

The idem field is the property you got for free. Transport relabels every Bool-slot in the record by the equivalence (inputs via from, outputs via to) and carries the proof along the same relabelling. Add ten more laws to Meet and it is still one transport.

World B — without univalence

Here is the world you already know, spelled out, because the contrast is the whole point.

The wall

There is no path Bool = Binary you can construct. refl only gives Bool = Bool; nothing builds a path between two distinct types. So transport Meet p boolMeet is unwritable: there is no p. All you have is the equivalence as plain function data, plus the round-trip laws, which are paths inside the types, not between them: 

toFrom : (a : Binary) -> to (from a) = a     -- a path inside Binary
fromTo : (x : Bool)   -> from (to x) = x     -- a path inside Bool

So you build it by hand, and re-prove every law

andB : Binary -> Binary -> Binary
andB a b = to (and (from a) (from b))

-- re-prove idempotence for Binary. Two ways, both fresh work:

-- option A: redo the case analysis on Binary's constructors
andBIdem : (a : Binary) -> andB a a = a
andBIdem one  = refl
andBIdem zero = refl

-- option B: port it through the equivalence, by hand
andBIdem a =
    andB a a
  = to (and (from a) (from a))    -- unfold andB
  = to (from a)                   -- by  ap to (andIdem (from a))
  = a                             -- by  toFrom a

Option B is transport done by hand: push into Bool with from, use the Bool proof, come back with to, tidy up with the round-trip law. It works. But it is bespoke for this one law, and you write another like it for every law and every representation. N laws times M representations, none reused. Univalence does this once, generically, for the whole bundle.

The deepest gap: one principle is not merely inconvenient, it is unprovable

"Any property of Bool is a property of Binary" fails as a theorem without univalence. A predicate that separates them:

P : Type -> Type
P A = (A = Bool)          -- "A is equal to Bool"

pBool : P Bool
pBool = refl              -- holds trivially

-- P Binary = (Binary = Bool) has NO inhabitant without univalence.
-- So P Bool holds and P Binary does not: P tells the two types apart.

So representation independence ("no property distinguishes equivalent types") is false in the old world. Univalence is exactly the axiom that flips it: it makes Binary = Bool inhabited, so that predicate stops separating them, and P Bool → P Binary becomes provable for every P at once. That is the thing no amount of by-hand porting can give you, because it is a statement about all predicates uniformly.

Does the result compute—yes in cubical, stuck in book-HoTT
The framing that ties it together. Mathematicians already treat isomorphic structures as interchangeable and move theorems between them freely. But in ordinary type theory that is an informal habit living outside the system. Univalence pulls it inside as a uniform, one-line operation, and cubical makes that operation actually run.
Two IOUs, tagged honestly, to be earned in later lessons: (1) quotient types without setoid hell (rationals, multisets, syntax up to renaming) come from higher inductive types; (2) function extensionality (equal-if-pointwise-equal) is another axiom in book-HoTT that cubical makes compute. Both are the same story as above: a payoff that only truly pays off once it computes.

Say it in your own words

Explain to a colleague what the Bool/Binary example shows about univalence, and what it costs to live without it. Three sentences.

Univalence turns the equivalence Bool ≃ Binary into a real equality Bool = Binary, so a single transport carries any structure and its proofs across for free, no matter how many laws it has. Without univalence there is no such path, so you re-prove every law by hand for each representation, and the general principle "every property transfers" is not even provable, because the predicate "= Bool" separates the two types. Cubical supplies the last piece: it makes that transport actually compute instead of getting stuck.
