The Two Equalities

Every proof you will read leans on two different notions of "equal" that look alike and behave nothing alike. This lesson pins down which is which, and why HoTT keeps them apart on purpose.

The win for this lesson You can say what ≡ and = each mean, point to the one rule that would collapse them, and explain why taking that shortcut would cost you both a working typechecker and univalence. That is why HoTT sits on intensional type theory, and why the rest of this course exists.

One fact, two statuses

Take addition on the naturals, defined the usual way by recursion on the first argument:

zero  + m = m
suc n + m = suc (n + m)

Now look at two facts that are both plainly true, and watch how differently the machine treats them.

-- 0 + n : the term matches the first clause and computes to n.
--         The typechecker sees they are the same. No proof needed.
0 + n  ≡  n        -- holds by computation

-- n + 0 : n is a variable, so no clause fires, and the term is stuck.
--         The checker does NOT see these as the same. You must prove it.
n + 0  =  n        -- needs a proof, by induction on n

Same mathematical truth, two completely different statuses. The difference is not about the numbers. It is about whether the definition of + makes the term reduce. That split is the whole subject of this lesson.

Naming them precisely

Following the precision we have been keeping, here are the two objects, kept strictly apart:

Judgmental a ≡ bPropositional a = b
What it isA judgment the typechecker makes. Not a type.A type. Its inhabitants are proofs, i.e. paths.
Who decides itThe machine, by reducing both sides to normal form.You, by building a term with refl, J, transport.
Can you hold one?No. It either holds or it does not; there is no evidence to pass around.Yes. A proof p : a = b is data you can name and use.
Other namesdefinitional equality, "computes to"the identity type, path, identification
Shorthand worth memorising: ≡ is what the machine already believes; = is what you can prove. Everything below is about the gap between those two.

Where they touch

They are separate, but not sealed off. Look at the one constructor of the identity type:

refl : a = a

Why is refl allowed to prove a = a, and also 0 + n = n? Because the checker first reduces both endpoints, and 0 + n reduces to n. So refl typechecks at 0 + n = n for free. The propositional layer is seeded by the judgmental one: whatever the machine already believes, refl hands you as a proof. Then J and transport grow that seed into everything else. What refl cannot reach, like n + 0 = n, you have to earn by induction.

The fork: intensional vs extensional

There is a tempting rule you could add. It is called equality reflection, and it says the two arrows are secretly one:

-- Equality reflection: if you can prove it, the machine believes it.
if   p : a = b     then     a ≡ b

A type theory with this rule is extensional. One without it is intensional: there, a proof p : a = b never upgrades into a ≡ b. The two stay in their lanes. This one rule is the entire fork in the road.

Extensional looks like a bargain. Function extensionality would come free: prove f and g agree everywhere, reflect it, done. So why does HoTT refuse it?

What the shortcut costs

Equality reflection charges two bills, and HoTT can pay neither.

Cost 1: typechecking stops being decidable To check a ≡ b, the machine now has to know whether some proof p : a = b exists. Searching for an arbitrary proof is undecidable. So the typechecker can no longer reliably say yes or no. Reduction was mechanical; proof search is not.
Cost 2: it forces UIP, which kills univalence Equality reflection implies UIP: every type becomes a set, the tower capped at rung 2. But Lesson 2 showed the universe must be a groupoid, because Bool = Bool has two distinct proofs. A theory that forces UIP cannot host univalence. The shortcut and the payoff are incompatible.

So HoTT stays intensional. It keeps ≡ and = apart, which keeps typechecking decidable and leaves room for a tall tower. The price is that extensionality is no longer free. You have to add it as real content.

A precision point you will care about. In bare intensional type theory, function extensionality is not false. It is independent: unprovable and unrefutable, consistent to assume either way, exactly the status representation independence had in Lesson 2. HoTT does not refute the extensional shortcut. It declines the reflection rule and instead earns extensionality as an axiom or a theorem in the propositional layer, where it does no damage to decidability.

Why this is the whole map

This fork is the reason the remaining lessons look the way they do. Because HoTT is intensional, it owes you extensionality as separate content:

  • Function extensionality (Lesson 9): an axiom in book-HoTT. It typechecks but does not compute, so proofs using it get stuck.
  • Univalence (Lesson 7): also an axiom in book-HoTT, and also stuck for the same reason.
  • Cubical (Lessons 10 to 11): the interval makes both of these into theorems that actually reduce. That is the reason cubical exists.

Hold onto the shape: intensional setting creates a bill, book-HoTT pays it with axioms that do not run, cubical pays it with theorems that do.

Check yourself

Answer from memory first. The click gives immediate feedback.

With + defined by recursion on its first argument, why is 0 + n ≡ n free while n + 0 = n must be earned?

0 + n matches the clause zero + m = m and computes to n, so the machine sees the equality and refl proves it. In n + 0 the variable n sits in the recursion slot, no clause fires, the term is stuck, and you must go by induction.

Adding equality reflection (making the theory extensional) has what unavoidable cost?

Reflection lets you conclude a ≡ b from any proof p : a = b, so the checker must search for proofs, which is undecidable. The same rule entails UIP, making every type a set, which contradicts the groupoid universe univalence needs.

Say it in your own words

A colleague asks: "Why not just make type theory extensional so function extensionality is free? Why does HoTT insist on keeping ≡ and = apart?" Answer in three sentences.

Judgmental equality ≡ is what the typechecker decides by computation, while propositional equality = is a type of proofs you build; extensional type theory adds equality reflection, which collapses the second into the first. That collapse makes typechecking undecidable, because the machine would have to search for proofs, and it forces UIP, which caps every type at the level of a set. HoTT refuses the shortcut so that typechecking stays decidable and the universe can be a groupoid, which is exactly what univalence needs, and it pays for extensionality separately, first with axioms and later, in cubical, with theorems that compute.
