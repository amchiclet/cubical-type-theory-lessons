Equivalence: the Right Notion of "Same"

"Has an inverse" is almost the right definition. The fix is small but crucial: being an equivalence must be a property, not extra data. Univalence depends on getting this exactly right.

The win for this lesson You can say why plain "isomorphism" needs repair in HoTT (it is not a mere proposition), give the clean repair (contractible fibres), and see why the fixed-up ≃ is exactly what the dictionary from Lesson 2 was, and what univalence will need.

The intuitive idea: isomorphism

You already know what "the same" should mean for two types: a function back and forth that undo each other.

-- f is an isomorphism (a "quasi-inverse") if there is g going back, with:
f : A -> B      g : B -> A
g ∘ f = id      -- go over and back on A: home
f ∘ g = id      -- go back and over on B: home

This is the honest intuition, and it is exactly the "dictionary" idea: f and g translate between A and B without losing anything. Nothing wrong with it as a picture. The trouble is in how you package it.

The catch: "has an inverse" is not a property

Write out "f has a quasi-inverse" as a type:

isIso f  =  Σ (g : B -> A). (g ∘ f = id) × (f ∘ g = id)

Here is the surprise: for a fixed f, this type is not a mere proposition. It can have many distinct inhabitants: different inverse-plus-homotopy packages that are not equal to each other. So "f has an inverse", packaged this way, is data, not a yes/no fact.

Why that is a problem We want to define A ≃ B = Σ (f : A → B). (f is an equivalence). If "f is an equivalence" were not a proposition, then two equivalences with the same underlying function could still differ, by their coherence data. That is junk: an equivalence should be determined by its function. We need "is an equivalence" to be a genuine property (a mere proposition), so that an equivalence is its function and nothing more.
Honesty flag: that isIso f is not a proposition is a real theorem, not a slogan. The type of quasi-inverses of a function is as complicated as the loops in the type (for the identity on A, roughly (x : A) → x = x), which is not yes/no. Proof: HoTT Book §4.1.

The fix: contractible fibres

The cleanest repair. First, the fibre of f over a point b : B is its preimages:

fibre f b  =  Σ (a : A). (f a = b)      -- "the points of A that f sends to b"

Then define:

isEquiv f  =  (b : B) -> isContr (fibre f b)
              "every b has exactly ONE preimage, up to a path"
equivalence:  every point of B has exactly one preimage

   A            B
   a₁ ────────▶ b₁      fibre f b₁ = { a₁ }     contractible ✓
   a₂ ────────▶ b₂      fibre f b₂ = { a₂ }     contractible ✓

NOT an equivalence:
   a₁ ─┐
   a₂ ─┴──────▶ b        fibre f b = { a₁, a₂ }  two preimages ✗
                 b'      fibre f b' = { }         no preimage  ✗

This is manifestly a proposition: isContr is a proposition (Lesson 3), and a Π of propositions is a proposition. So "is an equivalence" is now a genuine yes/no. And it is still logically the same as having an inverse: "exactly one preimage everywhere" is precisely what lets you define the inverse g (send each b to its unique preimage). Best of both: the iso idea, packaged as a property.

There are three standard good definitions, all propositions and all logically equivalent to "has a quasi-inverse": contractible fibres (above, the most visual); bi-invertible — a separate left inverse and right inverse, (Σ g. g∘f = id) × (Σ h. f∘h = id), the easiest to check is a proposition; and half-adjoint equivalence — one inverse with the two homotopies plus a coherence law linking them (the HoTT Book's default, §4.2). Pick whichever; they define the same ≃.

Why a property and not data? (the "why", concretely)

What the extra data looks like. On the circle, certify that id : S¹ → S¹ is an iso: give g = id and homotopies η, ε : id ∘ id = id. A homotopy id = id is (by funext) a choice of loop x = x at every point — and on S¹ you can pick "wind around n times" for any integer n. So there is a ℤ's worth of distinct witnesses that id is an iso: not yes/no, a whole ℤ of coherence choices. (For a set like Bool this collapses to nothing, which is why the problem only surfaces for higher types.)

Why that junk hurts, four concrete costs:

  • You can't say "the inverse." It would depend on which witness you picked. Uniqueness earns the definite article.
  • Coherence tracking. Every use of the equivalence would carry its witness and prove things respect it — setoid hell one floor up. A property carries nothing.
  • An equivalence wouldn't be its function. Two "equivalences" with the same f but different windings would count as different. Absurd for something meaning "f is invertible."
  • Univalence would miscount. A ≃ A would gain that spurious ℤ, so it couldn't match A = A, and (A = B) ≃ (A ≃ B) would fail.
What "well-behaved" means, concretely Because isEquiv f is a proposition, Σ (f). isEquiv f is the subtype of invertible functions: an equivalence is just a function (that happens to be invertible), and two equivalences are equal exactly when their functions are. This is the same subtype mechanic as Lesson 3's Σ (n:ℕ). n ≤ 10 = "just a number" — the proof component drops out of every comparison because it is a prop. That is the "attack surface": with data, comparing (f,w₁) = (g,w₂) needs f = g and w₁ = w₂; with a prop, the witnesses auto-agree and it collapses to f = g. Fewer moving parts, nothing to go wrong.
The category-theory reading: "the fibre is contractible" says "there exists a unique preimage" — a universal property. Uniqueness (contractibility) is prized for exactly the reason universal properties are: canonical, choice-free, transportable.

What an equivalence is, and why it is the right "same"

A ≃ B  =  Σ (f : A -> B). isEquiv f       -- isEquiv f is a PROPOSITION

So an equivalence is a function that is an equivalence, and being one adds no data. Two equivalences are equal exactly when their functions are: an equivalence is its function. That is the well-behaved notion of "same" for types. It is the precise version of the Lesson 2 dictionary, and it is what the next lesson pairs with identity:

The bridge to univalence Univalence will say (A =𝒰 B) ≃ (A ≃ B) — identity of types is equivalence of types. That statement only makes sense because ≃ is this careful, propositional notion. Had we used the naive iso (not a proposition), the two sides would not match and univalence would be malformed. The care taken here is exactly what lets univalence be stated at all.

Why is the naive "f has a two-sided inverse" a poor definition of equivalence in HoTT?

For a fixed f, "has a quasi-inverse" can have many distinct inhabitants, so it is data, not a yes/no property. We want "is an equivalence" to be a proposition, so an equivalence is determined by its function alone.

In the contractible-fibres definition, f is an equivalence exactly when:

isEquiv f = (b : B) → isContr (fibre f b): every b has a contractible fibre, i.e. exactly one preimage up to a path. Being a Π of propositions, this is itself a proposition.

Say it in your own words

Explain why HoTT replaces "has an inverse" with a fancier definition of equivalence, and what the replacement guarantees. Three sentences.

The naive "has a two-sided inverse" (quasi-inverse) is logically fine but is not a mere proposition: for a given f it can carry extra higher structure, so A ≃ B built from it would carry junk. HoTT instead defines "f is an equivalence" by a condition that is a proposition — for instance, every fibre of f is contractible (each b has exactly one preimage up to a path) — which is still logically equivalent to having an inverse. Because it is a proposition, "being an equivalence" is a genuine yes/no property, an equivalence is determined by its function, and the statement of univalence (A = B) ≃ (A ≃ B) comes out well-formed.
