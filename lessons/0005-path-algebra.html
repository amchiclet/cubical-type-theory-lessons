Path Algebra: ap, transport, and J

Three moves let you work with paths. Two are easy. The third, path induction, looks impossible until you see what it really says, and it is the engine that computes what "equal" even means.

The win for this lesson You can state path induction, say why proving the refl case is enough (it is not "every path is refl"), and watch it do real work: proving true ≠ false and computing the identity type of a whole structure. You will also see why that engine gets stuck in book-HoTT but runs in cubical.

Two easy moves first

You already met the idea of using a path. Two moves cover most of it.

-- ap: a function sends a path to a path. (Functoriality.)
ap : (f : A -> B) -> (a = a') -> (f a = f a')

-- transport: carry a point of the fibre P a along a path into P a'.
--            This is the "swapping" you already have a feel for.
transport : (P : A -> Type) -> (a = a') -> P a -> P a'

ap builds new paths from old ones. transport moves data across a path. Neither is mysterious. Both, it turns out, are special cases of the third move.

The move that looks impossible: path induction (J)

Here is the rule, based at a point a:

-- To prove a property C of EVERY endpoint y and EVERY path p : a = y,
-- it suffices to prove C at the single case (a, refl).
J : (C : (y : A) -> (a = y) -> Type)
    -> C a refl
    -> (y : A) (p : a = y) -> C y p

This should bother you. In Lesson 2 you saw paths that are visibly not refl: ua not : Bool = Bool flips the two values. On the circle there is a loop that goes around. So how can checking refl alone settle a claim about all paths?

The resolution J never says "all paths are refl." Look again: the endpoint y is free. J ranges over paths with one end pinned at a and the other end loose, that is, over the space Σ (y : A). a = y. And that space is contractible: its centre is (a, refl), and every other point (y, p) slides back to it. To prove a property of every point in a contractible space (a rung-0 type, Lesson 3), proving it at the centre is enough. That is the entire content of J: refl is the centre of the based path space, not the only path in it.

Another way to see it: count the constructors

The same fact from the angle that makes it feel like ordinary induction. Every inductive type gives one proof case per constructor, and a step case appears exactly when a constructor is recursive (takes an earlier copy of itself).

NAT    two constructors, suc RECURSES          ->  base + STEP
       0 --suc--> 1 --suc--> 2 --suc--> ...
       a chain; the step walks  n -> n+1

a = y  one constructor, refl does NOT recurse  ->  base only, NO step
              • y1
             /
   (a,refl) •───── • y2    refl is the lone generator;
             \             nothing to walk, so no step case
              • y3

suc : ℕ → ℕ takes a number, so it is recursive and earns a step case. refl : a = a takes no path, so it is not recursive and earns none. J is just "the identity type has one constructor, refl, so handle that one case." Same principle as natural-number induction, fewer constructors.

ap, transport, and sym are all J

This is the promise from the top of the lesson, made good. J is the general recursor; the easy moves are J with two blanks filled in: a motive (what to produce at the free endpoint y) and a refl-case (the value at the centre).

you wantmotive: produce at yvalue at refl
transport (carry x : P a)P yx itself
ap ff a = f yrefl : f a = f a
sym (reverse a path)y = arefl : a = a
J (general)any C(y, p)any d : C(a, refl)

Read the ap f row: you tell J "at endpoint y I want a proof f a = f y," and "at refl the answer is refl." J extends that one choice to every path, handing back f a = f b. So if you can read J, you have already read ap, transport, and sym: they differ only in those two columns.

Then why doesn't ua not break it?

Because ua not is a loop: both of its endpoints are pinned to Bool. J's contractible space has one free endpoint, so a loop never enters it as a fresh point to flatten. Contractibility does give you (Bool, refl) = (Bool, ua not) in the total space Σ (y). Bool = y, but the path witnessing that moves both components: its first part is a nontrivial Bool = Bool. So the two facts live in different types and never collide:

ua not ≠ refl            -- in  Bool =_U Bool   (endpoints pinned: a real loop)
(Bool, refl) = (Bool, ua not)   -- in  Σ (y). Bool = y  (endpoint free: holds)

J flattens the free-endpoint space. Loops keep both endpoints pinned, so they stay out of its reach. That is exactly why J is consistent with a universe that is a groupoid.

K: the axiom that would pin the endpoint

The loop discussion above is the exact hinge for axiom K (Streicher), J's stronger cousin. J keeps the endpoint free; K keeps it pinned and still lets you assume the loop is refl:

J : prove C(y, p) for all y and p : a = y    -- endpoint FREE   (handle (a, refl))
K : prove C(p)    for all loops  p : a = a    -- endpoint PINNED (handle refl)

Feed K the motive C(p) := (p = refl). Its refl-case is refl = refl, trivially true, so K hands you p = refl for every loop. That is UIP: every type is a set. J cannot do this, because it refuses to pin the endpoint, and that refusal is the entire difference between them.

So K is not newer or more advanced than J. It is an optional, stronger eliminator, and adding it caps the tower at rung 2. But ua not : Bool = Bool is a loop that is not refl, so K is simply false once univalence is in play. J stays; K is declined. That one choice is why HoTT has a tall tower at all.

J at work: what does "equal" mean for a type?

Now the payoff. J lets you compute the identity type of a structure. Start with the smallest case, and reconnect to something you asked about: proving things are genuinely different.

Define a family on Bool that tells the two values apart:

Code : Bool -> Type
Code true  = Top       -- the one-point type (inhabited)
Code false = Bottom    -- the empty type

Suppose someone hands you a path p : true = false. Transport along it:

transport Code p    : Code true -> Code false     -- that is  Top -> Bottom
transport Code p tt : Bottom                       -- but Bottom is empty!

So (true = false) -> Bottom, which is exactly true ≠ false. Notice what happened: you proved a disequality by swapping. You transported a distinguishing property along the hypothetical path and hit a contradiction. Disequality, the thing that keeps Bool from collapsing to a point, turns out to be an application of transport.

A precision point worth keeping: ¬(a = b) is always a mere proposition, a plain yes/no, even when a = b is a rich space full of paths. And recall the three statuses from Lesson 4: provable equality, provable disequality, and independent. true ≠ false is provable; ua not ≠ refl only becomes true once univalence is in play.

The same trick, run to completion: encode-decode

That Code family was a baby case of a general recipe. Give a two-argument version, and J proves that it is the identity type:

Code : Bool -> Bool -> Type
Code true  true  = Top      Code false false = Top
Code true  false = Bottom   Code false true  = Bottom

(x = y)  ≃  Code x y        -- the identity type of Bool, decoded

This is the encode-decode method: to find out what a path in a type really is, define a Code that mirrors the type's structure, then use J to prove the identity type is equivalent to it. The equivalence is built by path induction, because the forward map is "transport the reflexivity witness," exactly the move above.

The pattern: identity mirrors structure

Run this recipe on any structured type and the identity type comes out as a formula in the identity types of its parts:

((a, b) = (a', b'))   ≃   (a = a') × (b = b')            -- pairs
(f = g)               ≃   (x : A) -> f x = g x            -- functions (funext!)
((a, p) = (a', p'))   ≃   Σ (q : a = a'). transport q p = p'   -- dependent pairs

You have been reading formulas like these since Lesson 3 without naming them: isSet A = (x y : A) -> isProp (x = y) is identity nested inside identity. And the same recipe, run on the circle, is how the next lessons will prove π₁(S¹) = ℤ: the paths are the integers. Identity stops being only a tool for swapping and becomes an object you compute with.

Does the engine actually run? HoTT vs cubical

The proof principle J is the same in both systems. Whether it computes is not. This is the through-line from Lesson 4: the intensional setting owes you extensionality, and how the bill gets paid decides whether things run.

book-HoTTcubical
What a path isprimitive; identity type with J as a black boxa function I → A out of the interval
J isa primitive eliminatorderived from transport and hcomp
J on reflreduces to the base case definitionally (≡)equal to the base case only up to a path (=)
J on a ua/funext pathstuck: those are axioms, they do not reducecomputes: they are built from the interval, so transport walks along them
Canonicitybreaks once ua/funext are addedpreserved

So in book-HoTT, J is justified by the model but only actually fires on the syntactic refl. Feed it a path from univalence and the term jams. Cubical makes every path a real function transport can traverse, so J runs on all of them. The price: J's own reduction on refl is now only propositional. That trade is the reason cubical exists.

Honesty flag: the "only up to a path" behaviour is the CCHM / cubical Agda story. Some Cartesian cubical variants add a regularity condition that recovers a definitional J on refl. I am confident about CCHM; I would want to check a source before spelling out the regularity details.

Check yourself

Answer from memory first. The click gives immediate feedback.

Path induction proves a claim about all paths by checking only refl. Why is that enough?

J quantifies over paths with one free endpoint, that is, over Σ (y). a = y, which is contractible with centre (a, refl). Proving the property at the centre of a contractible space suffices. There are plenty of other paths (ua not, loop); they are just loops with pinned endpoints, outside the space J ranges over.

You apply J to a path built from univalence. What happens in book-HoTT versus cubical?

In book-HoTT J only reduces on the syntactic refl, and a univalence path is an axiom that does not compute, so J jams. In cubical every path is a function out of the interval and J is derived from transport, so it walks along the univalence path and computes.

Say it in your own words

Explain, in three sentences: why proving only the refl case suffices for path induction, and why that principle still runs on a ua-path in cubical but not in book-HoTT.

Path induction does not claim every path is refl; it quantifies over paths with one free endpoint, and that based path space is contractible with centre (a, refl), so proving the property at the centre is enough. In book-HoTT, J is a primitive rule that only reduces on the syntactic refl, so applied to a path from the univalence axiom, which does not compute, it gets stuck. In cubical, every path is a function out of the interval and J is derived from transport, so it walks along the ua-path and computes, at the cost that J's own reduction on refl holds only up to a path.
