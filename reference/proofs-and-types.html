Proofs, Dependency, Relevance

Three ideas that are easy to merge and worth keeping apart. HoTT needs all three, but they are independent knobs.

1. Proofs-as-objects (Curry-Howard)

The idea that a proof is a term and a proposition is a type. It is old and paper-friendly: proof theory already studies proofs (Gentzen's derivations) as objects, and Curry-Howard is the observation that those objects are lambda terms. This works with simple types, and on its own reaches only propositional logic (∧, ∨, →, no quantifiers, no equality). A proof assistant just mechanizes the checking of these proof-terms. Note: mechanical checking does not require this idea; HOL and set-theory provers check proofs that are meta-level derivations, not terms.

2. Dependency (dependent types)

A dependent type is a type that mentions a value: a family B : A → Type. Semantically this is just an indexed family of sets, so it lives comfortably on paper and in a proof-irrelevant world (ordinary set theory has it). Its contribution is expressive power: it lifts Curry-Howard from propositional logic to predicate logic. (x : A) → P x is "for all," Σ (x : A). P x is "there exists," and the identity type a = b is a dependent type. That last one is why HoTT cannot start without dependency.

The point that trips people up The value a type depends on is just a term, and under Curry-Howard proofs are terms too. So a type can depend on ordinary data (n in Vec A n) or on a proof. You usually meet dependency over data first, because indexing over a proof of a plain proposition is vacuous (all proofs are equal, so there is nothing to vary over). HoTT's move is to depend on proofs of equality (paths), which are not vacuous. That is exactly transport, and it is where dependency finally carries content.

3. Proof relevance

Whether two proofs of the same statement count as different. This is a separate knob from dependency. Dependent type theory plus UIP makes proofs of a proposition all equal (everything is a set); dependent type theory plus univalence makes proofs of type-equalities genuinely differ. Same dependent types, opposite relevance. So "proofs can be different" is not something dependency gives you; it is a choice you make on top.

What is UIP (the one you keep forgetting)

UIP — Uniqueness of Identity Proofs Any two proofs of a = b are themselves equal. Equivalently: every type is a set. It caps the h-level tower at rung 2, and it contradicts univalence: collapsing ua not and refl into one proof would force false = true. Its equivalent form is axiom K; Agda's --without-K turns it off.

How they stack for HoTT

You need all three, in order: (1) proofs-as-terms, so equality proofs are objects you can manipulate; (2) dependency, so the identity type a = b exists at all; then (3) drop UIP, so proofs of equality can differ. Univalence is what forces step 3 by handing type-equalities distinct proofs. See Lesson 2 for why that matters and Lesson 3 for the ruler (h-levels) that measures relevance.

Ask your teacher if any axis still blurs into another. The usual slip is treating dependency and relevance as the same thing; they are not.
